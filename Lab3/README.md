# Дизайн‑студія — консольний застосунок (TypeScript)

## Варіант та вимоги
**Варіант 8: Студія дизайну.**  
Потрібно реалізувати застосунок для студії, що надає комплексні дизайнерські послуги: від сайтів до приміщень.  
Функціонал має дозволяти:
- перегляд доступних послуг (звичайні та «під ключ»);
- перегляд портфоліо (усі виконані роботи);
- оформлення замовлення на послуги з каталогу;
- оформлення замовлення «під ключ»;
- розширення портфоліо (додавання нових робіт);
- розширення каталогу послуг (додавання нових послуг).

## Опис
Проєкт демонструє багаторівневу архітектуру з розділенням на **UI**, **BLL** та **DAL**.  
Реалізовано консольний інтерфейс для керування послугами, портфоліо та замовленнями.  
Використано **Repository**, **Unit of Work**, **DTO** і **mapping**.

## Структура проєкту
```
src/
  dal/        // доступ до даних (entities, repository, uow, db context)
  bll/        // бізнес-логіка (services, dto, mapping, validations)
  ui/         // консольний інтерфейс
  index.ts    // точка входу
uml/          // UML діаграми 
```

## Основний функціонал
- Перегляд послуг (звичайні / під ключ)
- Пошук послуг за назвою
- Перегляд портфоліо
- Додавання нової роботи у портфоліо
- Оформлення замовлення з каталогу
- Оформлення замовлення “під ключ”
- Перегляд усіх замовлень
- Перевірки даних (порожні поля, дублікати, позитивна ціна)

Після оформлення замовлення виконана робота автоматично додається у портфоліо.

## Запуск
```
npm install
npm run start
```

## Архітектура
**UI → BLL → DAL**

- **UI**: взаємодія з користувачем (консоль)
- **BLL**: бізнес‑логіка, DTO, перевірки, mapping
- **DAL**: сутності, репозиторії, UoW, контекст даних

# Відповіді на контрольні питання

1. **Що таке багаторівнева архітектура ПЗ та які переваги її використання?**  
   Це підхід, де система поділена на логічні рівні (UI, BLL, DAL). Переваги: краща підтримуваність, тестованість, гнучкість, ізоляція змін, повторне використання.

2. **Які існують шаблони проектування для організації шару доступу до сховищ даних?**  
   Найпоширеніші: Repository, Unit of Work, Data Mapper, DAO, Active Record.

3. **Поясніть ціль використання технології Entity Framework (EF) в архітектурі програмних систем.**  
   EF спрощує доступ до БД через ORM, зменшує ручний SQL, підтримує міграції та підвищує продуктивність розробки.

4. **Поясніть сутність шаблону Repository.**  
   Репозиторій інкапсулює доступ до даних для певного типу сутності і надає методи CRUD, приховуючи джерело даних.

5. **В чому полягає особливість використання generic репозиторію.**  
   Один універсальний репозиторій може обслуговувати різні типи сутностей, зменшуючи дублювання коду.

6. **Поясніть сутність шаблону Unit of Work.**  
   UoW координує роботу кількох репозиторіїв і забезпечує єдину точку збереження змін.

7. **Як пов’язані EF та шаблони Repository та Unit of Work?**  
   EF може виступати як внутрішня реалізація Repository та UoW (DbContext фактично є UoW).

8. **Поясніть сутність процесу mapping.**  
   Mapping — перетворення об’єктів між рівнями (Entity ↔ DTO), щоб уникнути прямої залежності між шарами.

9. **Наведіть шари та їх призначення в багаторівневій архітектурі ПЗ.**  
   UI — взаємодія з користувачем;  
   BLL — бізнес‑логіка;  
   DAL — доступ до даних.

10. **Опишіть властивості шарів багаторівневої архітектури ПЗ.**  
    Кожен шар має чітку відповідальність, залежить тільки від нижчих шарів, не містить чужої логіки.

11. **В чому полягає принцип ізоляції рівнів у багаторівневій архітектурі.**  
    Вищі шари не знають про внутрішню реалізацію нижчих (через інтерфейси та DTO), що дозволяє змінювати реалізацію без змін у всій системі.

12. **Яка користь від створення та використання сутностей (entities) на кожному рівні багаторівневої архітектури?**  
    Це знижує зв’язність, дозволяє змінювати структуру даних на одному рівні без впливу на інші, покращує безпеку і контроль даних.
