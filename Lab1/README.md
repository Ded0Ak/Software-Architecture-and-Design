# Лабораторна робота: 1, Варіант 8

## Загальна інформація про варіант
У цій лабораторній роботі реалізовано консольний застосунок, що моделює процес вивчення дисциплін:
- Основи програмування
- Об'єктно-орієнтоване програмування
- Алгоритми та структури даних

Застосунок працює з групами студентів, підгрупами для лабораторних занять, видами активностей дисциплін, умовами допуску до підсумкового контролю та подієвою моделлю змін стану.

## Що потрібно було реалізувати і що реалізовано
1. Модель дисциплін і активностей:
лекції, лабораторні, курсова робота, модульні контрольні, екзамен, залік (лише для Основ програмування).

2. Роботу з групами, студентами та підгрупами:
дисципліни вивчаються групами, лабораторні проводяться в підгрупах, підгрупа не може мати менше 10 студентів.

3. Перевірку технічної готовності:
для вивчення дисципліни кожен студент має мати комп'ютер або ноутбук.

4. Зміну активностей у межах дисципліни:
передбачено зміну набору активностей і обробку події зміни.

5. Обмеження за годинами:
на дисципліну виділяється не менше 64 аудиторних годин.

6. Обмеження за курсом:
- Основи програмування: лише 1 курс
- Алгоритми та структури даних: лише 2 курс
- ООП: 1-2 курс
- з 3 курсу і далі ці дисципліни недоступні

7. Унікальність вивчення дисципліни:
дисципліна в межах групи не може бути пройдена повторно.

8. Допуск до МКР та екзамену:
без семестрових робіт студент не допускається.

9. Автоматичний залік:
для Основ програмування залік виставляється автоматично.

10. Подієва модель:
реалізовано події для ключових змін стану (зміна активностей, зміна стану допуску) та їх демонстрацію в консолі.

## Запуск застосунку
1. Встановити залежності:
   npm install
2. Запустити застосунок:
   npx tsx src/app/main.ts

## Відповіді на питання

### 1. Особливості OOP: проблеми і типові рішення
OOP базується на інкапсуляції, абстракції, наслідуванні та поліморфізмі. Проблеми виникають, коли класи мають забагато відповідальностей, містять дублювання або тісно залежать один від одного. Типові рішення: чітко визначати межі відповідальності класів, приховувати внутрішній стан, працювати через стабільні контракти, надавати перевагу композиції над надмірним наслідуванням.

### 2. Що таке абстракція і що вона дає
Абстракція це виділення суттєвого і приховування деталей реалізації. Вона зменшує складність, робить модель предметної області зрозумілішою і дозволяє змінювати реалізацію без зміни зовнішнього контракту. Використовувати абстракцію варто там, де є повторювані ролі, спільна поведінка або очікується розвиток функціоналу.

### 3. Single Responsibility Principle
Суть SRP: клас має мати одну причину для змін. Типова проблема: один клас одночасно виконує бізнес-логіку, валідацію, логування і UI. Рішення: розділяти доменну логіку, прикладний сценарій, інфраструктурні аспекти та представлення даних на окремі компоненти.

### 4. Open-Closed Principle
Суть OCP: модулі мають бути відкриті для розширення і закриті для модифікації. Проблема: кожна нова вимога змушує редагувати стабільний код. Рішення: працювати через інтерфейси, стратегії, поліморфізм, конфігуровані правила та розширення через нові класи, а не через перепис базових.

### 5. Liskov Substitution Principle
Суть LSP: підтип має коректно замінювати базовий тип без зміни очікуваної поведінки. Проблема: підклас порушує контракт, наприклад змінює сенс методів або кидає неочікувані помилки. Рішення: формалізувати передумови і післяумови, не посилювати обмеження в підтипах, перевіряти замінність через тести контрактів.

### 6. Interface Segregation Principle
Суть ISP: краще мати кілька вузьких інтерфейсів, ніж один перевантажений. Проблема: клієнти змушені залежати від методів, які не використовують. Рішення: розбивати великі інтерфейси на рольові, мінімізувати поверхню API і залишати тільки релевантні операції для конкретних споживачів.

### 7. Dependency Inversion Principle
Суть DIP: залежати від абстракцій, а не від конкретних реалізацій. Проблема: високорівневий модуль жорстко прив'язаний до деталей, через що ускладнюються заміна і тестування. Рішення: вводити контракти, ін'єкцію залежностей, фабрики або композиційний корінь для збирання об'єктів.

### 8. Різниця між абстрактним класом і інтерфейсом
Інтерфейс описує контракт без стану екземпляра і без жорсткої прив'язки до ієрархії. Абстрактний клас може містити часткову реалізацію, спільний стан і захищені допоміжні методи. Інтерфейс обирають для ролей і слабої зв'язаності, абстрактний клас для повторного використання спільної поведінки в одній доменній лінійці.

### 9. DRY, YAGNI, KISS
DRY забороняє дублювання знань у коді. YAGNI закликає не реалізовувати те, що ще не потрібно. KISS вимагає простих і прозорих рішень. Типові проблеми: передчасне ускладнення, надмірна універсальність, копіювання логіки. Рішення: рефакторинг повторів, інкрементальна розробка, пріоритет простоти над складними конструкціями.

### 10. Cohesion і Coupling
Висока cohesion означає, що елементи модуля логічно пов'язані однією метою. Низька coupling означає мінімальну залежність між модулями. Проблема: зміна одного модуля викликає каскад змін в інших. Рішення: чіткі межі контекстів, стабільні контракти, поділ на рівні і мінімізація прямого доступу до внутрішнього стану інших модулів.

### 11. Inheritance with caution, композиція проти наслідування
Наслідування доцільне, коли дотримується істинне відношення is-a і зберігається контракт базового типу. Проблеми: крихкі ієрархії, порушення LSP, важка еволюція. Композиція краща, коли потрібна гнучка збірка поведінки, заміна реалізацій і повторне використання без жорсткої ієрархії.

### 12. Основні питання під час проєктування ПЗ
- Яка ключова бізнес-мета системи?
- Які інваріанти предметної області не можна порушувати?
- Де межі відповідальності модулів?
- Які сценарії зміни очікуються найчастіше?
- Як забезпечити тестованість і спостережуваність поведінки?
- Які обмеження продуктивності, надійності та безпеки?

### 13. Практики проєктування
Поширені практики: domain modeling, використання SOLID, рев'ю дизайну, ADR (Architectural Decision Records), рефакторинг малими кроками, тест-дизайн через сценарії, поділ на шари, event-driven підхід для відстеження змін стану, регулярна валідація архітектури відносно вимог.

### 14. Code standards і code conventions
Вони потрібні для єдиного стилю коду, кращої читабельності, швидшого рев'ю та зменшення помилок. Для C# це зазвичай офіційні рекомендації Microsoft, .NET coding conventions, аналізатори Roslyn, налаштування через EditorConfig, правила іменування, форматування, модифікатори доступу, організація просторів імен та файлів.

### 15. Duck Behavior
Duck Behavior це підхід, за якого важлива не формальна спорідненість типів, а наявність потрібної поведінки. Принцип використовується у поліморфних сценаріях, динамічній типізації та шаблоні Strategy. Ідея: якщо об'єкт підтримує потрібні операції, його можна використовувати в сценарії незалежно від конкретного класу.
