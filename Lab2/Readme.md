# Лабораторна робота №2 — Архітектура та шаблони

## Опис
Проєкт моделює процес вивчення університетських дисциплін із використанням шаблонів проектування:
- **Creational:** Factory Method
- **Behavioral:** Observer (через події)
- **Structural:** Composite

---

## Що потрібно було реалізувати і що реалізовано

### Вимоги завдання
1. Дисципліни: PB, OOP, ADS  
2. Активності: LEC, LAB, CP, MT, EX + PF тільки для PB  
3. Групи та підгрупи (мінімум 10 студентів у підгрупі)  
4. Наявність ПК/ноутбука для навчання  
5. Мінімум 64 аудиторних години  
6. Обмеження за курсами (PB=1, ADS=2, OOP=1-2, 3+ не можна)  
7. Дисципліна може вивчатись лише один раз  
8. Допуск до MT/EX тільки після семестрових робіт  
9. Автоматичний залік  
10. Заміна активностей та викладачів  
11. Викладач не може вести кілька дисциплін одночасно  
12. Керування через консоль  
13. Події (Observer)  
14. Використання шаблонів проектування  
15. Composite для Group/Subgroup  

### Реалізовано в проєкті
- **Factory Method:** `DisciplineFactory`, `UniversityDisciplineFactory`  
- **Observer:** `EventSubject/Observer/Subscription`, подія `workSubmitted`, підписка викладачів  
- **Composite:** `StudentComponent`, реалізація в `Group` і `Subgroup`  
- **Логіка допусків, годин, обмежень курсів**  
- **Консольне меню для всіх дій**  
- **Валідації домену через `DomainError`**

## Контрольні запитання (відповіді)

### 1. Що таке Code Smells? Які ви знаєте види? Наведіть приклади. Чому це погано?
**Code Smells** — це симптоми проблем у коді, які не є багами, але сигналізують про погану структуру.  
**Види:**  
- **Рівень застосунку:** god object, low cohesion  
- **Рівень класів:** large class, data class  
- **Рівень методів:** long method, duplicate code  
**Погано**, бо ускладнює підтримку, тестування й розширення.

### 2. Що таке spaghetti code? Чому це погана практика?
Spaghetti code — код без чіткої структури, з хаотичними залежностями.  
Погано, бо важко читати, тестувати й змінювати.

### 3. Які Code Smells належать до недоліків рівня застосування? Чому?
- God Object  
- Low Cohesion  
- Excessive Coupling  
Це проблеми на рівні архітектури, бо впливають на весь застосунок.

### 4. Які Code Smells належать до недоліків рівня класів? Чому?
- Large Class  
- Data Class  
- Inappropriate Intimacy  
Це проблеми всередині об’єктної моделі.

### 5. Які Code Smells належать до недоліків рівня методів? Чому?
- Long Method  
- Duplicate Code  
- Too Many Parameters  
Це ознаки поганої деталізації логіки.

### 6. Для чого потрібні шаблони проектування?
Для повторного використання перевірених рішень, спрощення підтримки та розширення.

### 7. Чи завжди потрібно використовувати багато шаблонів?
Ні. Шаблони потрібні лише коли вони покращують структуру — інакше це ускладнення.

### 8. Які існують групи шаблонів проектування?
- **Creational** — створення об’єктів  
- **Structural** — композиція  
- **Behavioral** — взаємодія  
Так поділяють за основною задачею.

### 9. Що таке Creational шаблони?
Шаблони для керування створенням об’єктів (Factory Method, Abstract Factory тощо).

### 10. Що таке Structural шаблони?
Шаблони для складання об’єктів у більші структури (Composite, Adapter).

### 11. Що таке Behavioral шаблони?
Шаблони для взаємодії між об’єктами (Observer, Strategy).

### 12. Про що говорить GRASP?
GRASP — принципи розподілу відповідальностей в ООП (Controller, Creator, Low Coupling тощо).

### 13. Чим відрізняються централізовані репозиторії від розподілених?
Subversion має один центральний сервер. Git — кожен має повну копію історії.

### 14. Яка структура Git репозиторію?
`.git/` містить об’єкти, refs, HEAD, індекс, історію.

### 15. Які види вказівників існують в Git?
- **Lightweight tags**
- **Annotated tags**
- **Branches**
- **HEAD**
Відрізняються призначенням і наявністю метаданих.

### 16. Що таке коміт? Яка його структура? Як зробити?
Коміт — знімок стану. Містить tree, parent(и), автора, дату, message.  
`git add .` → `git commit -m "msg"`

### 17. В яких статусах може перебувати файл репозиторія?
- untracked  
- modified  
- staged  
- committed

### 18. Які стани репозиторію існують і як пов’язані?
Working tree → Staging → Repository (commit)

### 19. Як можна переміщувати різні вказівники Git?
- `git branch -f`  
- `git reset`  
- `git checkout`  
Для зміни історії або переходу між станами.

### 20. Як зв’язати локальний репозиторій з віддаленим?
`git remote add origin URL`  
`git push -u origin main`  
`git pull` / `git fetch`

### 21. Які архітектурні стилі ви знаєте?
Клієнт‑сервер, багаторівнева, SOA, мікросервіси, компонентна, подійна, шина.

### 22. В чому полягає поєднання архітектурних стилів?
У використанні кількох стилів одночасно (наприклад, клієнт‑сервер + багаторівнева).

### 23. Архітектура клієнт‑сервер
Клієнт запитує ресурси, сервер обробляє і відповідає.

### 24. Компонентна архітектура
Система складається з незалежних компонентів з чіткими інтерфейсами.

### 25. Багаторівнева архітектура та N-рівнева архітектура
Розділення на рівні (UI, бізнес‑логіка, дані). N‑рівнева — більше ніж 3.

### 26. Шина повідомлень
Компоненти обмінюються повідомленнями через централізований bus.

### 27. Об’єктно-орієнтована архітектура
Система побудована навколо об’єктів і їх взаємодії.

### 28. Сервісно-орієнтована архітектура (SOA)
Система як набір сервісів, що взаємодіють через контракт.

### 29. Що варто реалізовувати як наскрізну функціональність?
Логування, безпека, аудит, кешування — бо вони стосуються багатьох модулів.
