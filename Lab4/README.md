# Software Architecture & Design — Lab 4 (WebUI)

## Опис завдання
Реалізовано WebAPI для управління **послугами, портфоліо та замовленнями** дизайн‑студії з рівнем представлення (UI).  
Система складається з шарів:
- **UI (HTML/JS)** — інтерфейс користувача
- **API (Express)** — приймає HTTP‑запити
- **BLL** — бізнес‑логіка
- **DAL** — збереження у JSON‑файл
- **InMemoryDbContext** — робота з даними в памʼяті з автозбереженням

### Реалізовано:
- CRUD для **Services / Portfolio / Orders**
- Перевірки на унікальність
- Автододавання замовлень у портфоліо
- Синхронізація портфоліо при зміні/видаленні замовлення
- PUT/DELETE
- UI кнопки для редагування/видалення
- Розділення за ролями (UI/API/BLL/DAL)
- Генерація окремих ID для кожної сутності

# Відповіді на контрольні питання (розширені)

### 1. Що таке API? Що таке WebAPI?
**API** — це набір правил, методів і форматів, за якими одна програма може взаємодіяти з іншою.  
API описує, **що можна викликати**, **які дані передати** і **що отримаємо у відповідь**.  
**WebAPI** — це API, доступне через HTTP/HTTPS. Воно дозволяє клієнтам (браузер, мобільний застосунок, інший сервіс) надсилати запити і отримувати дані у форматі JSON/XML.

---

### 2. Які задачі вирішує та в чому особливості рівня представлення PL?
Рівень представлення (**Presentation Layer**) відповідає за:
- відображення інформації користувачу (UI/UX),
- збір вводу,
- просту валідацію.  
Його особливість — **мінімум бізнес‑логіки**. Основна логіка повинна бути у BLL, а PL лише показує дані.

---

### 3. Які типові шаблони проектування існують для реалізації PL?
Найпоширеніші шаблони:
- **MVC** (Model‑View‑Controller)
- **MVP** (Model‑View‑Presenter)
- **MVVM** (Model‑View‑ViewModel)  
Також існують: **PAC**, **HMVC**, **MVU** — рідше використовуються.

---

### 4. В чому особливість та як реалізується шаблон MVC?
MVC розділяє систему на 3 частини:
- **Model** — дані та бізнес‑логіка
- **View** — відображення
- **Controller** — обробляє запити, викликає Model, передає дані у View  
Схема: **Request → Controller → Model → View → Response**.

---

### 5. В чому особливість та як реалізується шаблон MVP?
У MVP:
- **View** пасивна (не містить логіки),
- **Presenter** містить всю логіку, керує View.  
View і Presenter спілкуються через інтерфейс, що полегшує тестування.

---

### 6. В чому особливість та як реалізується шаблон MVVM?
MVVM використовує **data binding** між View і ViewModel.  
ViewModel містить стан та логіку, а View автоматично оновлюється, коли дані змінюються.

---

### 7. Опишіть шлях обробки запиту від браузера клієнта до рівня представлення (MVC).
1. Браузер надсилає HTTP‑запит  
2. Маршрут визначає потрібний **Controller**  
3. Controller викликає **Model**  
4. Model повертає дані  
5. Controller передає дані у **View**  
6. View формує HTML  
7. HTML повертається у браузер  

---

### 8. Чим відрізняється ASP.NET MVC та ASP.NET WebAPI?
- **ASP.NET MVC** повертає HTML‑сторінки (UI).  
- **ASP.NET WebAPI** повертає дані (JSON/XML), UI відсутній.

---

### 9. Чим відрізняється WebAPI як рівень представлення від інших клієнт‑серверних застосувань?
WebAPI не показує UI, а лише повертає дані.  
Воно може обслуговувати різні клієнти: веб, мобільні додатки, інші сервіси.  
У класичних клієнт‑серверних системах клієнт часто є жорстко прив’язаним.

---

### 10. Які види серверів ви знаєте?
- **Web server**  
- **Application server**  
- **Database server**  
- **File server**  
- **Mail server**  
- **Proxy / DNS сервери**  

---

### 11. Що таке веб сервер і які його основні задачі? Приклади
Веб‑сервер:
- приймає HTTP‑запити,
- віддає статичні файли (HTML, CSS, JS, images).  
Приклади: **Nginx, Apache, IIS**.

---

### 12. Що таке сервер застосувань і які його основні задачі? Приклади
Сервер застосувань:
- виконує бізнес‑логіку,
- обробляє API‑запити,
- формує відповіді.  
Приклади: **Node.js (Express), ASP.NET Kestrel, Tomcat, Spring Boot**.

---

### 13. Поясніть протокол HTTP та його роботу
HTTP — це протокол “запит‑відповідь”.  
Клієнт надсилає: метод + URL + headers + body.  
Сервер відповідає: статус‑код + headers + body.  
HTTP є **stateless**: сервер не пам’ятає стан клієнта між запитами.

---

### 14. В чому різниця між GET/POST/PUT/DELETE?
- **GET** — читання ресурсу  
- **POST** — створення нового  
- **PUT** — повне оновлення  
- **DELETE** — видалення  
Це основні CRUD‑операції в REST.

---

### 15. Поясніть основні групи статус кодів HTTP
- **1xx** — інформаційні  
- **2xx** — успіх (200, 201)  
- **3xx** — перенаправлення  
- **4xx** — помилки клієнта (404, 400)  
- **5xx** — помилки сервера (500)

---

### 16. В чому полягає особливість реалізації DI для ASP.NET MVC чи WebAPI?
DI дозволяє:
- передавати залежності через конструктор,
- легко міняти реалізації (моки/тести),
- централізовано керувати життєвим циклом сервісів.  

---

### 17. Поясніть особливості використання Postman, Fiddler та подібних інструментів
- **Postman** — створення/тестування HTTP‑запитів (GET/POST/PUT/DELETE).  
- **Fiddler** — перехоплення і аналіз HTTP‑трафіку між клієнтом і сервером.  
Обидва корисні для дебагу API.

---

### 18. В чому різниця між веб застосуванням та веб сервісом?
- **Веб застосування** має UI для людини.  
- **Веб сервіс** повертає дані для інших програм.  

---

### 19. Що таке REST?
REST — архітектурний стиль, де все — **ресурси**, які мають URL, а дії виконуються стандартними HTTP‑методами.

---

### 20. В чому відмінність REST від RPC та SOAP?
**REST** працює з ресурсами і стандартними HTTP методами.  
**RPC** викликає методи як функції (наприклад `CreateOrder`).  
**SOAP** базується на XML і строгих контрактах (WSDL), складніший у використанні.  
REST простіший і легше інтегрується у сучасні системи.

---

### 21. Як побудована маршутизація в ASP.NET WebAPI та як вона працює?
Маршрутизація визначає, який URL відповідає конкретному контролеру.  
Може бути:
- **атрибутна** (`[Route("api/orders")]`)
- **конвенційна** (templates)

---

### 22. Як визначається, який контроллер та який метод будуть виконуватися у відповідь на запит?
Визначається за:
- URL маршрутом
- HTTP методом
- назвою action  
Наприклад: `GET /api/orders` → `OrdersController.Get()`.
